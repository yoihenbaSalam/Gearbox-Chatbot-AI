Raw PDF Store — Every Uploaded Document Persisted Before Processing
Before any extraction or transformation begins, every uploaded PDF is written verbatim to the raw zone of the data lake under a deterministic path constructed from the document hash, gearbox model, and upload timestamp. This ensures the original source is always recoverable — if a chunking strategy changes, an embedding model is retrained, or a processing error corrupts a downstream collection, the raw file can be re-ingested from scratch without asking anyone to re-upload it. The file registry table in PostgreSQL tracks every ingested file, its processing status, and its Kafka event ID.
raw/ gearbox-manuals/ G900/ G900_Technical_Manual_Rev4_sha256abc123_20250614.pdf G1200/ G1200_Manual_Rev2_sha256def456_20250614.pdf lessons-learned/ LessonsLearned_PlantA_2024_sha256ghi789_20250614.pdf # PostgreSQL file registry INSERT INTO file_registry ( file_id, original_name, lake_path, sha256_hash, gearbox_model, doc_type, sensitivity, status, kafka_event_id, uploaded_at ) VALUES ( 'FILE-0041', 'G900_Technical_Manual_Rev4.pdf', 'raw/gearbox-manuals/G900/G900_Technical_Manual_Rev4_sha256abc123.pdf', 'abc123...', 'G-900', 'TECHNICAL_MANUAL', 'INTERNAL', 'PENDING', 'evt-9912', NOW() )
Zone Architecture — Raw, Processed, and Curated Layers
The data lake is split into three zones so that each pipeline stage reads from and writes to a clearly-owned area. The raw zone is immutable after write — no process is permitted to modify or delete a file once stored there. The processed zone holds extracted artefacts (text JSON, table CSV, PNG images). The curated zone holds the final chunk objects with their metadata envelopes, ready for embedding. MinIO is the recommended self-hosted S3-compatible store for on-premise plant deployments; AWS S3 or Azure Blob for cloud deployments.

Deduplication & Version Control
Every file is SHA-256 hashed before storage. If the same hash already exists in the registry, the upload is flagged as a duplicate and no new lake file is written — the existing record is linked to the new upload request. If a new version of an existing manual arrives (same model, different revision), the previous version is moved to the archive zone and a new raw entry is created, triggering a full re-ingestion pipeline run for that document only. This keeps the vector collections current without a full re-index of all documents.

Pipeline Orchestration — Every Stage Produces and Consumes Kafka Topics
Kafka acts as the nervous system of the entire ETL pipeline. Each processing stage consumes from an upstream topic and produces to a downstream topic — this decouples stages so that a slow embedding step cannot back-pressure the extraction step, retries happen automatically via consumer group offsets, and every event is persisted with a timestamp for the monitoring layer to replay or audit. A Dead Letter Queue (DLQ) topic captures any event that fails after three retry attempts, with the full payload and error trace preserved for investigation.

Idempotency & At-Least-Once Delivery
Each Kafka consumer uses the file_id as an idempotency key checked against the file_registry table before processing. If a file_id has already been processed to the COMPLETE state, the event is acknowledged and skipped without re-processing. This allows safe Kafka topic replay for monitoring or re-indexing scenarios without creating duplicate chunks or duplicate Neo4j nodes.

Primary Path — PyMuPDF Text-Layer Extraction with Quality Gate
PyMuPDF (fitz) reads the text layer of each page and returns text blocks with bounding-box coordinates. After extraction, a quality gate checks the character-per-page ratio and the proportion of garbled Unicode characters. If a page yields fewer than 50 characters or more than 15% replacement characters (U+FFFD), the quality gate classifies that page as extraction-failed and emits a page-level OCR request event onto the Kafka ocr.required topic — only the failing pages are routed to PaddleOCR, not the entire document, preserving throughput.
for page in doc: text = page.get_text("blocks") char_count = sum(len(b[4]) for b in text if b[6]==0) garbage_ratio = text.count("\ufffd") / max(char_count, 1) if char_count < 50 or garbage_ratio > 0.15: # Route to PaddleOCR fallback producer.produce("ocr.required", { "file_id": file_id, "page_no": page.number }) else: write_to_processed_zone(text, page.number)
PaddleOCR Fallback — Page-Level Rasterise and Recognise
When a page fails the quality gate, it is rasterised to a 300 DPI PNG using PyMuPDF's get_pixmap() and passed to PaddleOCR's PP-OCRv4 model, which runs a two-stage pipeline: a text detection model identifies text region bounding boxes, and a recognition model reads the characters within each box. PaddleOCR is chosen over Tesseract because it handles rotated text, curved labels on mechanical diagrams, and mixed-script documents significantly better — all common in industrial gearbox catalogues. The recognised text output is merged back into the page's extraction result and written to the processed zone alongside the primary-extracted pages.
from paddleocr import PaddleOCR ocr = PaddleOCR( use_angle_cls=True, # handles rotated text on diagram labels lang="en", use_gpu=True # GPU strongly recommended for throughput ) # Rasterise failing page at 300 DPI pixmap = page.get_pixmap(dpi=300) img_arr = np.frombuffer(pixmap.samples, dtype=np.uint8) img_arr = img_arr.reshape(pixmap.height, pixmap.width, pixmap.n) result = ocr.ocr(img_arr, cls=True) # result → [ [[x,y],text,confidence], ... ] ocr_text = " ".join([line[1][0] for block in result for line in block])
ℹ PaddleOCR also processes image blocks inside text-parseable pages — this extracts any text embedded within gearbox diagrams (callout labels, part-number stamps, dimension annotations) that PyMuPDF's text layer does not capture because they are rasterised into the image data.

Semantic Chunking — Topic Boundary Detection via Embedding Similarity
Unlike recursive character splitting which cuts at fixed token counts, semantic chunking embeds each sentence with a lightweight embedding model and splits only when the cosine similarity between adjacent sentence embeddings drops below a configurable threshold. This ensures each chunk covers exactly one coherent topic — a bearing replacement procedure is never split mid-step, and a torque specification sentence always remains with the paragraph that names the bolt it applies to. The threshold of 0.75 is empirically calibrated for technical maintenance text; lower values produce larger, more context-rich chunks, higher values produce finer-grained topic isolation.
from langchain_experimental.text_splitter import SemanticChunker from langchain_community.embeddings import HuggingFaceEmbeddings # Use a fast BERT encoder for similarity decisions (not the full BGE-M3) embed_model = HuggingFaceEmbeddings(model_name="BAAI/bge-small-en-v1.5") chunker = SemanticChunker( embeddings=embed_model, breakpoint_threshold_type="percentile", # or "standard_deviation" breakpoint_threshold_amount=75 # split when similarity drops to 75th pct ) chunks = chunker.create_documents([page_text])
✓ Gap 3 resolved — semantic chunking with document-type-aware override rules below
Document-Type Override Rules — BOM, P&T Tables, SOP Steps
Semantic similarity alone is not sufficient for all document types. Certain structured formats require hard boundaries that override the similarity threshold — a BOM row must never be merged with an adjacent row, a pressure-temperature table must be kept whole, and an SOP step must stay intact. These rules are implemented as pre-chunking document-type classifiers that intercept the text before it reaches the semantic splitter.

Why BGE-M3 over text-embedding-3-large
BGE-M3 produces both a 1024-dimensional dense vector and a sparse BM25-style vector in a single forward pass. This directly satisfies Gap 1 — hybrid BM25 search for exact part number matching (G-900, API 6D, SKF 6205) — without a separate keyword index. text-embedding-3-large produces dense vectors only, requiring an additional BM25 pipeline. BGE-M3 is therefore the technically superior choice for this architecture and is confirmed as the primary model.
Fine-tuning path
BGE-M3 can be domain-adapted using the FlagEmbedding library with a contrastive learning approach. Collect (query, positive_chunk, hard_negative_chunk) triplets from technician feedback — validated query-response pairs from the L7 feedback loop are a natural source. Fine-tuning for 3–5 epochs on ~5,000 triplets typically yields 8–15% recall improvement on domain-specific queries. The fine-tuned checkpoint replaces the base model in the embedding worker with no schema changes required in Milvus — only a re-index of existing vectors.
from FlagEmbedding import BGEM3FlagModel # Production inference model = BGEM3FlagModel('BAAI/bge-m3', use_fp16=True) output = model.encode( [chunk_text], return_dense=True, return_sparse=True, # ← enables hybrid search in Milvus max_length=512 ) dense_vec = output['dense_vecs'][0] # 1024d float32 sparse_vec = output['lexical_weights'] # dict → SPARSE_FLOAT_VECTOR

When to upgrade to this model
text-embedding-3-large outperforms BGE-M3 on documents exceeding 2,000 tokens per section, on cross-lingual retrieval across 5+ languages simultaneously, and on open-domain benchmarks (MTEB). If post-launch RAGAS faithfulness scores consistently fall below 90% on long-context queries and fine-tuning BGE-M3 does not close the gap, switching to text-embedding-3-large for the text_chunks collection is the justified escalation path. The table_chunks and image_vectors collections are unaffected — their models remain BGE-M3 and CLIP respectively.
Matryoshka dimension flexibility
text-embedding-3-large supports output truncation to 256d, 512d, 1024d, or 3072d via the dimensions parameter. Use 1024d in Milvus to keep the collection schema compatible with a BGE-M3 baseline so that a model switch does not require a Milvus collection rebuild — only a re-embedding run.
Fine-tuning path
OpenAI's fine-tuning API supports embedding model adaptation via contrastive training on (query, positive, negative) pairs. This is a cloud-only operation at current API capability — use only if vendor lock-in is acceptable and the performance gap justifies the cost and dependency.
⚠ API dependency — OpenAI outages halt embedding. Requires separate BM25 layer for hybrid search. Use only if BGE-M3 RAGAS targets are not met after fine-tuning.

Why CLIP is non-negotiable
Your 50–100 gearbox diagrams per manual are invisible to BGE-M3 — text encoders cannot process pixel data. OpenCLIP ViT-H-14 maps both image pixels and text descriptions into a shared 768-dimensional space, enabling text-to-image retrieval: "show me the G-900 bearing housing assembly" returns Figure 4.2 even if its caption reads only "Bearing assembly" because the visual content of the diagram aligns with the query in the shared CLIP space.
Fine-tuning path
OpenCLIP supports domain fine-tuning using contrastive image-text pairs. Collect (image, descriptive_caption) pairs from your manual figures — captions written by your engineering team are ideal. Fine-tuning for 10–20 epochs on 500+ image-caption pairs on a single GPU significantly improves recall for industrial machinery diagram retrieval compared to the general-purpose LAION-trained weights.

Hybrid Search — Dense ANN + Sparse BM25 via Milvus 2.4 AnnSearchRequest
The hybrid search mode combines the dense cosine similarity vector from BGE-M3 with its sparse lexical output, merging results via Reciprocal Rank Fusion. This resolves Gap 1: a technician querying "G-900 torque for housing bolt M12" benefits from exact BM25 match on "G-900" and "M12" combined with semantic retrieval of the surrounding torque specification context — something neither pure vector nor pure keyword search achieves alone.
from pymilvus import AnnSearchRequest, RRFRanker # Both dense and sparse from same BGE-M3 forward pass dense_req = AnnSearchRequest([dense_vec], "vector", {"metric_type":"COSINE","params":{"ef":200}}, limit=10) sparse_req = AnnSearchRequest([sparse_vec], "sparse_vector", {"metric_type":"IP"}, limit=10) # RBAC filter applied at search time — sensitivity scalar field expr = "sensitivity in ['PUBLIC','INTERNAL']" # Technician role results = col.hybrid_search( [dense_req, sparse_req], rerank=RRFRanker(k=60), limit=5, expr=expr, output_fields=["chunk_id","text","section","page","sensitivity","doc_type"] )

Table Path — Dual Write to Milvus and PostgreSQL
Every table row is serialised to natural language and embedded into table_chunks for semantic retrieval, and simultaneously written as a structured row to the PostgreSQL specs table where the NL-to-SQL path can retrieve it with exact arithmetic precision. This dual-write ensures that "what is the maximum torque for housing bolt M12 on G-900?" can be answered either semantically (Milvus, approximate match) or exactly (PostgreSQL, value = 120, unit = Nm, model = G-900, bolt = M12).

NER Pipeline — How Neo4j Nodes Are Created from Chunk Text
After semantic chunking and before embedding, each chunk passes through a Named Entity Recognition step that extracts Gearbox Model IDs, Part IDs, Error Codes, Technician references, and sensitivity-relevant terms. Each entity becomes a Neo4j node, and a DESCRIBES or MENTIONS relationship links that DocumentChunk to every entity it references. This bridge — the milvus_id on the DocumentChunk node — is what allows the orchestrator to take a Milvus search result and immediately expand it into a 2-hop graph neighbourhood without a separate lookup, enabling GraphRAG context enrichment.
entities = ner_model.extract(chunk.text) # → {"models":["G-900"], "parts":["BRG-HS-001","Seal-TypeX"], "errors":["ERR-022"]} for part_id in entities["parts"]: neo4j.run(""" MERGE (c:DocumentChunk {chunk_id: $cid}) MERGE (p:Part {id: $pid}) MERGE (c)-[:MENTIONS]->(p) """, cid=chunk.chunk_id, pid=part_id) # After Milvus insert: write milvus_id back to DocumentChunk node neo4j.run("MATCH (c:DocumentChunk {chunk_id:$cid}) SET c.milvus_id=$mid", cid=chunk.chunk_id, mid=milvus_result.primary_keys[0])

Authentication — Azure AD SSO with JWT Token Propagation
All users authenticate through Azure Active Directory Single Sign-On. On successful login, the Azure AD token is exchanged for a signed internal JWT containing the user's role (TECHNICIAN, ENGINEER, or ADMIN), their plant assignment, and their sensitivity clearance level. This JWT is attached to every API request and validated by the retrieval gateway before any Milvus or Neo4j query is constructed. Token expiry is set to 8 hours (one shift), with silent refresh supported for active sessions.

Golden Dataset Construction and Continuous Evaluation
A 200-question golden dataset is constructed from three sources: 80 questions derived directly from the technical manuals by engineers (ground truth = specific manual page), 80 questions from historical maintenance tickets with known resolutions, and 40 adversarial questions designed to test exact part number retrieval and RBAC boundary enforcement. RAGAS evaluation runs automatically in the Airflow DAG after every embedding model update, chunking strategy change, or Milvus index rebuild — results are written to the monitoring database and surfaced in the ETL dashboard.

Pipeline Processing History — Per-Document Audit Trail
Every Kafka topic event is consumed by a monitoring worker that writes a processing_log row to PostgreSQL. This creates a complete, queryable audit trail for every document: which stage it reached, how long each stage took, whether OCR fallback was triggered, how many chunks were created, how many Milvus inserts succeeded, and whether the RAGAS evaluation run for that ingestion batch passed all targets. Engineers can replay any failed document's Kafka events from the stored offset without data loss.
