Vectorless + Tri-Store
The Librarian
Milvus (Vector)
Handles all procedural, descriptive, and diagnostic queries — everything a technician would phrase in natural language. "What does it mean if the oil smells burnt?" Only Milvus can retrieve the answer from an unstructured PDF paragraph.
The Architect
Neo4j (Graph)
Owns every structural relationship — what is inside what, what depends on what, who is linked to who. Verifies that a retrieved torque value is for the right gearbox model before the LLM sees it. Prevents "correct value, wrong asset" hallucinations.
The Accountant
PostgreSQL (SQL)
Owns all time-series, numeric, and historical data. Required for the predictive/prescriptive roadmap — you cannot run linear regression or a random forest model on a graph or vector database. Building Tri-Store now is building the prediction foundation.

Three Critical Challenges — Token Cost · Overkill Argument · Scalability
Challenge 1 · Token Economics
The Hidden Tax
Ingestion cost · Cypher hallucination · context flooding
Vectorless ingestion cost
To build a fully vectorless graph, an LLM must read every sentence and extract (Subject)-[Predicate]→(Object) triples. For a 500-page gearbox manual, this means millions of tokens of LLM processing just to build the database — before a single query is answered. This is an enormous upfront tax with no clear ceiling as the manual library grows.
Cypher hallucination tax
Generating complex Neo4j Cypher queries via LLM requires long few-shot prompts — sending 10–20 example Cypher queries in every user message to guide the model. This inflates chatbot token count on every single query, regardless of whether the user asked a simple or complex question.
Hybrid cost profile
Creating BGE-M3 embeddings is effectively free ($0.0001 per 1k tokens). The LlamaIndex Small-to-Big approach sends 1 Parent Summary (200t) + 3 Leaf Nodes (~900t) + SQL metric (50t) = ~1,150 tokens per query versus ~10,000 for flat RAG or a Cypher few-shot prompt chain.
Vectorless per query: ~8,000–15,000 tokens (few-shot Cypher + graph context) Hybrid per query: ~1,150 tokens (1 parent + 3 leaves + SQL fact) Savings: ~88% token reduction
Challenge 2 · The Overkill Argument
Is it justified?
Vectorless overkill · Tri-Store insurance · industrial precision
Is vectorless overkill? Yes — for prose
Using a Knowledge Graph to store a paragraph about "how to clean a gearbox" is overkill. Graphs are designed for entities and relationships, not unstructured prose. Forcing prose into a graph produces a very expensive, slow version of a search engine — with none of the semantic recall benefits of an actual vector store.
Is Tri-Store overkill? No — for industrial AI
In gearbox asset management, precision is binary: you are either right, or the asset fails. A single incorrect torque value can lead to a $250,000 catastrophic failure. The "overkill" of three databases is insurance — you are paying for the certainty that the AI knows which gearbox, which section, and which specific part it is discussing before it answers.
The prediction roadmap argument
Predictive and prescriptive queries (Q46–50) require SQL time-series data for linear regression, random forest models, and MTBF calculations. You cannot run these on a graph or vector database. Starting with Tri-Store now is not overengineering — it is building the foundation for the prediction module that is already on your roadmap.
Challenge 3 · Tuning & Scalability
Future-proofing
Schema lock · modular updates · reranker tuning
Vectorless schema lock problem
If you decide to track a new data type — for example, "weather conditions during failure" or "operator shift at time of fault" — you must re-process all existing documents through the entity extraction pipeline to identify and write those new relationships into the graph. This is a massive re-ingestion effort that grows proportionally with your document library size.
Hybrid modular update advantage
Swapping the embedding model from BGE-M3 to a fine-tuned domain model takes an afternoon — run a re-embedding job on existing chunks, no schema changes in Milvus or Neo4j required. The Reranker (L5) weights can be adjusted independently to shift what is considered more important (live sensor data versus manual text) without touching any other component.
Fine-tune feedback flywheel
Every validated technician interaction generates a training triplet for BGE-M3 fine-tuning. After ~5,000 triplets, a domain-adapted checkpoint is deployed with 8–15% recall improvement. Vectorless systems have no equivalent continuous improvement loop — the graph schema is static between re-ingestion events.

STAGE 0A · STORAGE
Data Lake
& Raw File Store
MinIO / S3 · file registry · 4-zone
Raw PDF Store — Every Uploaded Document Persisted Before Processing
Before any extraction or transformation begins, every uploaded PDF is written verbatim to the raw zone of the data lake under a deterministic path constructed from the document hash, gearbox model, and upload timestamp. This ensures the original source is always recoverable — if a chunking strategy changes, an embedding model is retrained, or a processing error corrupts a downstream collection, the raw file can be re-ingested from scratch without re-uploading. The PostgreSQL file_registry table tracks every ingested file, its processing status, and its Kafka event ID.
raw/gearbox-manuals/G900/G900_Technical_Manual_Rev4_sha256abc123_20250614.pdf raw/lessons-learned/LessonsLearned_PlantA_2024_sha256def456_20250614.pdf # file_registry row { file_id, original_name, lake_path, sha256_hash, gearbox_model, doc_type, sensitivity, status, kafka_event_id, uploaded_at }
Zone Architecture — Raw (immutable) → Processed → Curated → Archive
Raw zone is write-once, never modified after upload. Processed zone holds extracted artefacts per page (text_blocks.json, tables.csv, images/*.png). Curated zone holds final chunk objects with full metadata envelopes, ready for embedding. Archive zone stores superseded manual versions for 7-year regulatory retention. SHA-256 deduplication prevents re-ingestion of identical files; new manual versions move prior versions to archive and trigger selective re-indexing of only that document's chunks.
STAGE 0B · EVENT BUS
Kafka
Event Bus

Pipeline Orchestration — Every Stage Produces and Consumes Kafka Topics
Kafka decouples stages so a slow embedding step cannot back-pressure extraction. Every event is persisted with a timestamp, enabling monitoring replay and audit. STAGE 1 · EXTRACT
Extraction
+ OCR Fallback
PyMuPDF primary · PaddleOCR PP-OCRv4 fallback
Primary Path — PyMuPDF with Per-Page Quality Gate
PyMuPDF extracts text blocks with bounding-box coordinates (x0, y0, x1, y1). A quality gate checks character density and garbled Unicode ratio per page. Pages yielding fewer than 50 characters or more than 15% replacement characters are emitted to an ocr.required Kafka sub-topic — only those pages route to PaddleOCR, preserving throughput on cleanly-parsed pages.
if char_count < 50 or garbage_ratio > 0.15: producer.produce("ocr.required", {page_no: page.number}) else: write_to_processed_zone(text_blocks, page.number)
PaddleOCR Fallback — Rasterise at 300 DPI, PP-OCRv4 Two-Stage
Failing pages are rasterised to PNG at 300 DPI and passed to PaddleOCR's PP-OCRv4 (detection + recognition). PaddleOCR also extracts callout labels embedded inside gearbox diagrams — part-number stamps, dimension annotations, assembly callouts — that PyMuPDF's text layer does not capture because they are rasterised into the image data. This makes every label in every technical diagram retrievable as text.
Metadata Envelope — doc_type · asset_class · sensitivity · page_number on every object

STAGE 1.5 · HIERARCHY
LlamaIndex
PageIndex Build
HierarchicalNodeParser · GPT-4o-mini · Parent summarisation
Structure Extraction — LLM Scans PDF to Build a PageIndex from Table of Contents
A GPT-4o-mini call scans the extracted text to identify the document's Table of Contents and creates a Parent Node for every Chapter and Section. For example, "Section 4.2: Bearing Assembly" becomes a Parent Node with a unique section_id. This is the "Upfront Tax" — a one-time cost at ingest time that pays dividends by eliminating structural ambiguity at query time. Without this pass, a vector search for "bearing torque" might retrieve a correct value from the wrong gearbox model's section — a silent, dangerous hallucination.
# Parent Node (Section level) ParentNode { node_id: "G900-S4.2", title: "Section 4.2: Bearing Assembly", model: "G-900", summary: "200-token LLM-generated summary of Section 4.2...", children: ["G900-S4.2-L001", "G900-S4.2-L002", ...] # leaf IDs }
Parent Summarisation — 200-Token Section Summaries via GPT-4o-mini
Every Parent Node is summarised into a 200-token node using GPT-4o-mini. This summary is stored in Neo4j as a Section node property and is what gets retrieved during the "Small-to-Big" expansion step at query time — when a Leaf Node's parent_id is resolved, the LLM receives the Leaf's specific fact plus its Parent's summary context, giving it structural awareness without flooding the context window with the entire section's text.
Cross-Linking — Leaf Nodes → Milvus, Hierarchy → Neo4j (Vectorless Path)
After leaf chunking, each Leaf Node is stored in Milvus (vector retrieval path) and simultaneously linked to its Parent Node in Neo4j via a BELONGS_TO relationship. This creates a parallel "vectorless" navigation path — if Milvus vector search fails, is ambiguous, or returns low-confidence results, the orchestrator can navigate the Neo4j document tree directly to find the correct section. This is the hybrid architecture: vector retrieval and structural graph retrieval operating in parallel, each acting as fallback for the other.

→ CONSUMED BY: Chatbot Stage L4 Hierarchy Route
STAGE 2 · CHUNKING
Semantic
Chunking
BGE-M3 similarity · doc-type override rules
Semantic Chunking — Topic Boundary Detection via Embedding Similarity
Unlike recursive character splitting which cuts at fixed token counts, semantic chunking embeds each sentence and splits only when cosine similarity between adjacent sentences drops below the 75th percentile threshold. Each chunk covers exactly one coherent topic — a bearing replacement procedure is never split mid-step, and a torque specification sentence always remains with the paragraph that names the bolt it applies to.
chunker = SemanticChunker( embeddings=HuggingFaceEmbeddings("BAAI/bge-small-en-v1.5"), breakpoint_threshold_type="percentile", breakpoint_threshold_amount=75 )

STAGE 3 · EMBEDDING MODELS — BGE-M3 (recommended) · text-embedding-3-large (alternative) · OpenCLIP (images) · Fine-tuning paths
RECOMMENDED · BERT-TYPE
BGE-M3
1024d dense + sparse · open-source · self-hosted
Why BGE-M3
Produces 1024d dense vector and a sparse BM25-style vector in a single forward pass — directly satisfying hybrid search for exact part numbers (G-900, SKF 6205, API 6D) without a separate keyword index. Confirmed primary model.
Fine-tuning path
FlagEmbedding contrastive training on (query, positive_chunk, hard_negative) triplets from the L7 feedback loop. 3–5 epochs on ~5,000 triplets yields 8–15% recall improvement with no Milvus schema changes — only a re-embedding run.
model = BGEM3FlagModel('BAAI/bge-m3', use_fp16=True) output = model.encode([chunk_text], return_dense=True, return_sparse=True) dense_vec = output['dense_vecs'][0] # 1024d sparse_vec = output['lexical_weights'] # SPARSE_FLOAT_VECTOR
JUSTIFIED ALTERNATIVE · GPT-TYPE
text-embedding-3-large
1536–3072d · OpenAI API · Matryoshka dims
When to upgrade
Outperforms BGE-M3 on documents >2,000 tokens per section and on cross-lingual retrieval. If post-launch RAGAS faithfulness scores fall below 90% and fine-tuning BGE-M3 does not close the gap, switch text_chunks to this model. Output 1024d via Matryoshka for Milvus schema compatibility — no collection rebuild needed, only re-embedding.
⚠ Requires separate BM25 layer for hybrid search. API outage halts embedding. Use only if BGE-M3 RAGAS targets unmet after fine-tuning.

HIERARCHICAL HYBRID — Our System
Vector + Vectorless
in parallel
Mutual fallback · structural awareness + semantic recall
How the hybrid works
Milvus retrieves Leaf chunks semantically. The parent_id on each Leaf triggers LlamaIndex RecursiveRetriever to fetch the Section summary from Neo4j. Neo4j's BOM graph verifies part membership before the LLM generates a response. If Milvus confidence is below threshold, the orchestrator falls back to Neo4j vectorless document tree navigation.
Vectorless fallback trigger
When top-k Milvus results all score below a similarity threshold of 0.65, LangGraph routes to the Neo4j document tree: the user can ask "show me Section 7" and the AI navigates via the Section nodes, returning the 200-token parent summary without any vector lookup. This makes the system functional even when embeddings are cold or low-confidence.

L1 · ENTRY GATE
Redis
Semantic Cache
sub-30ms · cosine sim > 0.95
Vector Embedding Match
Every incoming technician query is immediately converted into a dense vector and compared against a cache of previously resolved issues using cosine similarity. This happens before any database is touched, ensuring repeated or semantically similar questions are answered in under 30 milliseconds without triggering any downstream retrieval pipeline.
Gate Logic — Similarity > 0.95 Returns Instantly
If the similarity score between the incoming query vector and a cached result exceeds 0.95, the system returns the cached answer immediately and the full pipeline terminates here — no LLM call, no database query, no orchestration overhead. If the score falls below 0.95, the query is classified as a cache miss and forwarded to the pre-processor with its raw text intact.
IF cosine_sim(query_vec, cache_vec) > 0.95 → RETURN cached_response # <30ms, zero DB cost ELSE → FORWARD to L2 Pre-Processor # full pipeline continues

Unit Conversion — Imperial to Metric
When a field technician submits a query containing imperial measurements — such as "the shaft is vibrating at 0.5 inches per second" — the pre-processor uses a regex-matched unit mapping table combined with a lightweight LLM call to convert the value to the metric equivalent stored in the database. This prevents zero-result SQL failures caused by unit system mismatches, which are one of the most common silent failure modes in industrial AI deployments.
"0.5 inches/sec" → 12.7 mm/s "10 PSI" → 0.69 Bar "120°F" → 48.9°C "3000 RPM at load" → RPM_sensor = 3000, context = load
Entity Linking — Colloquial Names to Database IDs
Technicians refer to gearbox assets by shorthand names, site nicknames, or partial identifiers that do not match database primary keys. The pre-processor resolves "GB-402", "the big mill gearbox", or "unit 4 in Plant 7" to their canonical asset identifier in the PostgreSQL schema, ensuring every downstream query references the correct row without requiring the technician to know the internal data model.
"GB-402" → Asset_ID = 00402 "the bearing on line 3" → Part_ID = BRG-L3-HS "Sarah's last job" → Tech_ID = T-0047
Technical Slang Normalization
Field language such as "main shaft shaking," "gearbox running hot," or "weird noise from the housing" is classified and mapped to structured diagnostic terms that match the vocabulary used in sensor logs, maintenance records, and technical manuals, giving the orchestrator layer a clean, unambiguous signal to route against all three stores.

Vectorless Fallback Decision
After Milvus returns top-k results, their similarity scores are inspected. If all scores fall below 0.65, the orchestrator sets a low_confidence flag and routes to Neo4j's document tree navigation path — returning Section summaries without a vector lookup. This prevents the LLM from generating an answer from low-quality, potentially wrong-model chunks. The technician receives a structured manual summary rather than a hallucinated specific value.

L5 · FILTER
Context Fusion
& Reranker
BGE-Reranker-v2-m3 · Reciprocal Rank Fusion
RRF Merges Four Retrieval Worlds into One Ranked List
Results from PostgreSQL rows, Milvus Leaf chunks, Neo4j graph facts, and LlamaIndex Parent summaries all arrive simultaneously but in incomparable ranking spaces. RRF normalises them into a single unified list: RRF_score(d) = Σ 1/(k + rank_i(d)) where k=60, summed across all four retrieval sources. BGE-Reranker-v2-m3 then scores every (query, candidate) pair jointly to discard low-relevance items before they consume LLM tokens.
Critical application: a live PostgreSQL sensor reading showing "bearing is 80°C" correctly ranks above a 2019 manual excerpt stating "maximum temperature 75°C" because the SQL source is more recent and directly addresses the named asset.

If SQL data is missing for a specific asset, the LLM states: "I cannot find live sensor data for GB-402, but according to the G-900 manual the general limit is 85°C." If Milvus returns low-confidence results and vectorless path activates, the LLM states the structural source: "Based on Section 7 of the G-900 manual (browsed via document structure)." The system never generates a confident specific value from an ambiguous source.

Every validated interaction produces a (query, validated_chunk, hard_negative_chunk) triplet stored in a fine-tuning dataset. When the dataset reaches ~5,000 triplets, a FlagEmbedding fine-tuning run is triggered via Airflow, producing a domain-adapted BGE-M3 checkpoint. The new checkpoint replaces the base model in the embedding worker with no Milvus schema changes — only a re-embedding run on existing chunks.
