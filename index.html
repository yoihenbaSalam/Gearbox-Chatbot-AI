Vector Embedding Match
Every incoming technician query is immediately converted into a dense vector and compared against a cache of previously resolved issues using cosine similarity. This happens before any database is touched, ensuring repeated or semantically similar questions are answered in under 30 milliseconds without triggering any downstream retrieval pipeline.
Gate Logic — Similarity > 0.95 Returns Instantly
If the similarity score between the incoming query vector and a cached result exceeds 0.95, the system returns the cached answer immediately and the full pipeline terminates here — no LLM call, no database query, no orchestration overhead. If the score falls below 0.95, the query is classified as a cache miss and forwarded to the pre-processor with its raw text intact.
IF cosine_sim(query_vec, cache_vec) > 0.95 → RETURN cached_response # <30ms, zero DB cost ELSE → FORWARD to L2 Pre-Processor # full pipeline continues

Unit Conversion — Imperial to Metric
When a field technician submits a query containing imperial measurements — such as "the shaft is vibrating at 0.5 inches per second" — the pre-processor uses a regex-matched unit mapping table combined with a lightweight LLM call to convert the value to the metric equivalent stored in the database. This prevents zero-result SQL failures caused by unit system mismatches, which are one of the most common silent failure modes in industrial AI deployments.
"0.5 inches/sec" → 12.7 mm/s "10 PSI" → 0.69 Bar "120°F" → 48.9°C "3000 RPM at load" → RPM_sensor = 3000, context = load
Entity Linking — Colloquial Names to Database IDs
Technicians refer to gearbox assets by shorthand names, site nicknames, or partial identifiers that do not match database primary keys. The pre-processor resolves "GB-402", "the big mill gearbox", or "unit 4 in Plant 7" to their canonical asset identifier in the PostgreSQL schema, ensuring every downstream query references the correct row without requiring the technician to know the internal data model.
"GB-402" → Asset_ID = 00402 "the bearing on line 3" → Part_ID = BRG-L3-HS "Sarah's last job" → Tech_ID = T-0047
Technical Slang Normalization
Field language such as "main shaft shaking," "gearbox running hot," or "weird noise from the housing" is classified and mapped to structured diagnostic terms that match the vocabulary used in sensor logs, maintenance records, and technical manuals, giving the orchestrator layer a clean, unambiguous signal to route against all three stores.
L3 · DIRECTOR
Orchestrator
& Router
LangGraph Supervisor · Pydantic
SQL Trigger → PostgreSQL Route
When the Pydantic entity extractor detects statistical intent — keywords such as "how many," "average," "last serviced," "overdue," "count," or a specific date range — the orchestrator classifies the query as a metrics query and schedules a natural language to SQL translation against the PostgreSQL maintenance log and sensor telemetry tables.
Example: "How many gearboxes in Plant 7 have skipped their 6-month oil change?" → Routed to Postgres → NL-to-SQL → SELECT COUNT(*) FROM assets WHERE plant_id=7 AND next_service < NOW() → Output: "12 gearboxes are overdue"
Vector Trigger → Milvus Route
When the query contains procedural or instructional intent — phrases like "how to," "torque setting," "correct procedure," "step by step," "safety protocol," or references to specific model numbers alongside technical parameters — the orchestrator routes to Milvus to perform a top-k=5 semantic chunk retrieval across the ingested PDF manuals, lessons-learned reports, and troubleshooting guides.
Example: "What is the correct torque for the housing bolts on the G-900 series?" → Milvus vector search across G-900 Manual collection → Output: "120 Nm — page 45, G-900 Series Technical Manual, Rev. 4"
Graph Trigger → Neo4j Route + asyncio.gather()
When the query implies a dependency, impact, or relationship — phrases such as "affects which," "linked to," "what else will fail," or entity names like bearing, seal, shaft, or housing alongside causal language — the orchestrator activates Neo4j for a 2-hop neighbourhood search. All three store queries are fired simultaneously via asyncio.gather(), eliminating the 200–500ms sequential latency that would otherwise compound across stores.
Example: "If Seal Type-X runs out, which gearbox models are unrepairable next week?" → Neo4j Cypher → MATCH (s:Seal {type:'Type-X'})<-[:REQUIRES]-(g:Gearbox) RETURN g.model_name → Output: "Models G-900 and G-1200 will be affected as they use that specific seal"

Reciprocal Rank Fusion — Three Worlds, One Ranked List
The system receives up to 5 document chunks from Milvus, 3 result rows from PostgreSQL, and a subgraph from Neo4j — each ranked within its own retrieval system but incomparable to one another. RRF normalises these into a single unified ranking by computing a harmonic score based on each result's position across stores, ensuring that a highly-ranked SQL row and a highly-ranked manual excerpt can be compared and ordered before reaching the language model.
RRF_score(d) = Σ 1 / (k + rank_i(d)) where k=60 Summed across SQL rank, Vector rank, and Graph rank → Single ordered list handed to cross-encoder for final scoring
BGE Cross-Encoder — Joint Pair Scoring Discards Irrelevant Results
After RRF, the BGE-Reranker-v2-m3 cross-encoder scores every (query, candidate) pair jointly, reading both together rather than independently. This catches cases where a result looks superficially relevant by keyword but is semantically misaligned with the technician's actual intent, and it discards those results before they consume LLM context window tokens. A 2019 manual excerpt stating "maximum temperature 75°C" is correctly ranked below a live PostgreSQL sensor reading showing the bearing is currently at 80°C.

Redis Write-Back — Query-Response Pair Injected into Cache
When a technician clicks "This helped" on a response, the system immediately re-embeds the original query and stores the validated response in Redis with a confidence score of 1.0 and a time-to-live of 30 days. This means any semantically similar question asked within the next month — from any terminal in the plant — will be answered instantly from cache without touching any of the three retrieval stores, compounding speed improvements over time as more resolutions are validated.
{ "query_vector": [...], "response": "12mm/s exceeds 8mm/s threshold...", "validated_by": "Tech_ID_T-0047", "confidence": 1.0, "ttl_days": 30 }
Neo4j Edge Creation — Validated Solution Node Written to Graph
Simultaneously, the orchestrator writes a new VALIDATED_SOLUTION relationship node into Neo4j, connecting the specific gearbox asset, the diagnosed symptom, and the resolution text with the validating technician's ID and the timestamp. This means the next time any technician queries a vibration issue on any gearbox in the G-900 family, the 2-hop neighbourhood search will surface this field-proven resolution as a first-class result ranked above unvalidated manual excerpts, allowing the system's knowledge to grow organically from real plant experience.
MERGE (sol:Solution {id:'SOL-8821'}) MERGE (a)-[:HAS_SYMPTOM]->(s) MERGE (s)-[:RESOLVED_BY { validated_by: 'T-0047', date: '2025-06-14' }]->(sol)
